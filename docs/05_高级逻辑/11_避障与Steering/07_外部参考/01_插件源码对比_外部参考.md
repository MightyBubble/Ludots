# 避障与Steering 插件源码对比 外部参考

# 1 项目概述

## 1.1 项目定位与要解决的问题

本文档对比三类 Unity 侧实现，目的是提取可复用的算法语义与工程骨架，用于 Ludots 的避障/steering/群体模块设计。

对比对象：

- ProjectDawn：navigation / localavoidance / crowds（偏数据导向与 job 化）
- SteeringAI：可配置 steering pipeline（偏行为组合与可编辑配置）
- AStar：网格寻路 + penalty/islands + 局部避障思路（偏工程化工具集）

## 1.2 版本与关键依赖

本仓库不绑定具体版本号；源码位置以 Unity 工程目录结构描述（Packages/Assets），避免绑定本机绝对路径。

- ProjectDawn：`Packages/com.projectdawn.navigation`、`Packages/com.projectdawn.navigation.crowds`
- SteeringAI：`Packages/com.ondrejvaic.steeringai`
- AStar：`Assets/Athomield/AStar`

# 2 核心能力矩阵

## 2.1 能力点与映射表

| 能力点 | ProjectDawn | SteeringAI | AStar | 对 Ludots 的映射 | 风险与成本 |
|---|---|---|---|---|---|
| 近场避障求解 | Sonar/ORCA 语义 | 行为组合可替代 | 局部工具/思路 | Non‑Cullable 求解器模块 | 需隔离 Unity 依赖 |
| 群体引导 | crowds/flow-field | pipeline 可做近似 | penalty/islands | Cullable 的场/管线 | 需分帧迭代与预算 |
| 配置形态 | 代码/数据结构为主 | 配置中心化 | 工具集 + 配置 | token/ID + registry | 禁止反射类型名 |
| 性能导向 | SoA/Native 容器 | 视实现而定 | 视实现而定 | 固定容量 + workUnits | dropped/退化必须显式 |

# 3 适用性评估

## 3.1 适配成本与风险

1. 禁止复用 Unity.Entities/Jobs/NavMesh/Physics 外壳：只能抄算法语义与数据布局思想。  
2. Ludots 以离散 tick 为裁决真源：任何依赖 wall-clock 的逻辑必须改造。  
3. 空间候选必须来自空间服务门面（Span + Dropped）：插件的邻域采样必须接入该契约。  

## 3.2 与现有实现差异

- 空间查询接口与 dropped 口径：`docs/03_基础服务/06_空间服务/02_接口规范/01_空间查询接口_接口规范.md`  
- 多频率策略参考实现（Physics2D targetHz 分发）：`src/Core/Engine/Physics2D/DiscreteRateTickDistributor.cs`  

# 4 结论与建议

## 4.1 是否引入

不直接引入插件代码；仅复用算法语义与工程形态，并通过 Ludots 的接口契约重写。

1. 在项目内固化约束：`docs/05_高级逻辑/11_避障与Steering/04_裁决条款/01_更新频率与退化策略_裁决条款.md`
