# 4 插件对比（摘要）

本节为“对比摘要”，用于快速决策与定位吸收点；单插件细节请阅读对应的外部参考正文。

## 4.1 对比对象

- AnyPath：DOTS 算法库风格（Graph/Heuristic/EdgeMod/Processor + 可复用 workspace）
- A* Pathfinding Project（com.arongranberg.astar）：Unity 导航全家桶（多图/更新管线/层级图/后处理/避障）

## 4.2 能力矩阵与映射

| 能力点 | AnyPath | A* Pathfinding Project | 对 Ludots 的映射 | 风险与成本 |
|---|---|---|---|---|
| A*/Dijkstra | 支持 | 支持且成熟 | 已有 NodeGraphPathService；关注预算/统计与并行模型 | 整包引入收益低 |
| 策略扩展（过滤/改权/后处理） | 强（EdgeMod/Processor） | 中（配置+Modifiers） | 对齐 TraversalPolicy/Overlay + Processor | 需统一版本化与审计 |
| 动态更新管线 | 弱（偏图内能力） | 强（WorkItem/安全点） | 需要在 GraphWorld/Overlay 写阶段落地 | 不照搬锁，走阶段化调度 |
| 层级图/连通性加速 | 弱 | 强（HierarchicalGraph） | 作为 L0/L1 加速与一致性维护参照 | 数据结构重排成本高 |
| streaming/corridor | 不提供 | 非直接提供 | Ludots 必须自研（AOI+Chunk+LoadedView） | 误用外部库会失败 |
| Unity 工具链 | 有（示例/CodeGen） | 很强（Editor/扫描） | 不作为主线；工具链后置 | 强 Unity 绑定 |

## 4.3 结论

- 不整包引入 AnyPath 或 A* Pathfinding Project 作为 Ludots 主导航方案。
- AnyPath：吸收其接口组织方式（Graph/Policy/Workspace/Processor）与“复用工作内存”的工程习惯。
- A* Pathfinding Project：吸收其工程化边界（请求管线/批处理回收点/更新安全点/层级图维护思路）。
