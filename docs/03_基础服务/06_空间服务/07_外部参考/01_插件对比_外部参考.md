# 插件对比 外部参考

# 1 项目概述

## 1.1 项目定位与要解决的问题

本外部参考对比两类 Unity 生态方案，用于为空间服务建立选型语境，但不作为项目口径真源：

1. ECS Spatial Partitioning：强调按 cell 分桶、粗裁剪与分帧更新的工程范式。
2. GimmeDOTSGeometry：强调几何算法与空间结构（KD/R*/Octree 等）的工具箱范式。

## 1.2 版本与关键依赖

由于本项目不直接依赖 Unity，该对比仅提炼数据形态与裁决条款，不复刻 Unity API。
外部拆解入口（仓库内拷贝）：

- `docs/arch-guide/external/unity-ecs-spatial-partitioning/`
- `docs/arch-guide/external/GimmeDOTSGeometry/`

# 2 核心能力矩阵

## 2.1 能力点与映射表

| 能力点 | 外部方案支持情况 | 对 Ludots 的映射 | 风险与成本 |
|---|---|---|---|
| 只处理局部（按 cell/chunk） | Spatial Partitioning 强 | GridBuckets + time-slice | 需要自行实现 buckets 与指标 |
| 可见性粗裁剪 | Spatial Partitioning 强 | VisibleCells/VisibleChunks | 需要相机输入数据化与共享资源 |
| 视野外降频 | Spatial Partitioning 强 | tick-rate + time-slice | 必须以逻辑 tick 为真源 |
| 几何/构网算法 | GimmeDOTSGeometry 强 | 导航/地形离线构建工具箱 | 脱 Unity 迁移成本高 |
| 运行时确定性 | 两者都非核心目标 | 由空间服务裁决与验收保证 | 需要稳定序与浮点策略 |

# 3 适用性评估

## 3.1 适配成本与风险

1. 直接复刻 Unity API 是高风险：应只迁移分桶数据形态与裁决条款。
2. 若 AOI/Chunk 的 LoadedChunks 真源未统一，任何索引优化都会被加载不一致抵消。

## 3.2 与现有实现差异

差异以空间服务对齐报告为准：

- `docs/03_基础服务/06_空间服务/05_对齐报告/01_空间服务_对齐报告.md`

# 4 结论与建议

## 4.1 是否引入

不直接引入外部插件；仅吸收其可迁移的范式与数据形态。

## 4.2 后续动作

1. 将分桶 + time-slice + 指标可观测落地为项目裁决条款与验收用例。
2. 若未来需要构网/几何算法，再评估引入独立算法库或在本项目实现必要子集。
