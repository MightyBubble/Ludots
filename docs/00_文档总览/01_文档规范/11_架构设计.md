# 架构设计 标题与必选章节

# 1 标题规范

一级标题固定：

- `# {系统/模块} 架构设计`

# 2 必选章节

- `# 1 设计概述`
  - `## 1.1 本文档定义`
  - `## 1.2 设计目标`
  - `## 1.3 设计思路`
- `# 2 功能总览`
  - `## 2.1 术语表`
  - `## 2.2 功能导图`
  - `## 2.3 架构图`
  - `## 2.4 关联依赖`
- `# 3 业务设计`
  - `## 3.1 业务用例与边界`
  - `## 3.2 业务主流程`
  - `## 3.3 关键场景与异常分支`
- `# 4 数据模型`
  - `## 4.1 概念模型`
  - `## 4.2 数据结构与不变量`
  - `## 4.3 生命周期/状态机`
- `# 5 落地方式`
  - `## 5.1 模块划分与职责`
  - `## 5.2 关键接口与契约`
  - `## 5.3 运行时关键路径与预算点`
- `# 6 与其他模块的职责切分`
  - `## 6.1 切分结论`
  - `## 6.2 为什么如此`
  - `## 6.3 影响范围`
- `# 7 当前代码现状`
  - `## 7.1 现状入口`
  - `## 7.2 差距清单`
  - `## 7.3 迁移策略与风险`
- `# 8 验收条款`

裁决：

- 必须在“本文档定义 / 设计目标”中写明“边界”与“非目标”，防止后续文档无限膨胀。
- 默认不做向后兼容与静默 fallback：破坏性变更必须通过迁移/工具/编译期或加载期强约束解决，运行时保持单一真实来源并 fail-fast。
- “业务设计”必须至少包含 1 张可读的主流程图；关键场景与异常分支必须有对应流程表达。
- 正文不得贴大段实现代码：单个代码片段不超过 30 行；需要说明时优先用结构图/流程图/表格/伪代码，并在“现状入口”给出代码路径。
- “验收条款”必须可判定（不写口号），至少 3 条；每条必须给出验证方法与证据入口（测试/指标/文件路径）。

# 3 可选章节

按需选用。

- `# 替代方案对比`
- `# 性能与预算`
- `# 线程安全与并行策略`
- `# 配置与版本策略`
- `# 失败策略与诊断信息`
- `# 对齐报告与测试用例链接`
