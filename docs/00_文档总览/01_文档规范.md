# 文档规范

# 1 元信息（Front-Matter）

所有文档必须使用 YAML front-matter，放在文件首部，起止用 `---` 包裹。

## 1.1 必填字段

```yaml
---
文档类型: 总览入口
创建日期: 2026-02-05
最后更新: 2026-02-05
维护人: X28技术团队
文档版本: v0.1
适用范围: {模块} - {子系统} - {主题}
状态: 草案
---
```

字段口径：

- 文档类型：从受控枚举中选 1 个（见 1.3）。
- 创建日期：首次创建日期，不再回写。
- 最后更新：每次内容实质更新时改动。
- 维护人：团队或岗位，不写个人网名。
- 文档版本：语义版本；草案阶段从 v0.1 起。
- 适用范围：用“层级 - 模块 - 子主题”描述边界，必须能回答“属于谁/覆盖什么”。
- 状态：用于阅读决策与执行优先级（见 1.2）。

## 1.2 状态枚举

- 草案：允许大改结构；不作为唯一真源引用。
- 审阅中：结构基本稳定；等待裁决/评审结论。
- 已裁决：裁决条款已定稿；冲突时以本文为准。
- 已实现：主要方案已落地到代码；允许小幅修订与补充验收条款。
- 已废弃：保留历史入口；必须在正文首段指向替代文档。
- 兼容入口：仅用于保持历史外链稳定；正文只保留跳转说明。

## 1.3 文档类型枚举（9 种）

采用“单一标签”方案：每篇文档只允许一个 `文档类型`。

| 类型标签 | 定义 | 典型内容 |
|---|---|---|
| 架构设计 | 系统边界、模块关系、决策取舍 | 为什么这样设计、替代方案、风险与迁移 |
| 接口规范 | 对外 API 形状与调用约定 | 签名、参数表、错误码、线程安全性 |
| 配置结构 | JSON/YAML/二进制配置格式定义 | 示例、字段表、校验规则、版本策略 |
| 裁决条款 | 争议点 tie-break 与强制约束 | 稳定序、容量上限、禁止行为、fail-fast |
| 对齐报告 | Design vs Code 的审计记录 | 差异表、风险等级、行动项 |
| 测试用例 | 可复现的验收/回归场景 | 前置条件、步骤、预期结果、自动化状态 |
| 外部参考 | 第三方库分析与可迁移结论 | 能力矩阵、适用性评估、迁移成本 |
| 工具设计 | 工具链/离线管线/CLI 的设计与产物契约 | 输入输出、产物格式、版本化、安全性、命令与示例 |
| 总览入口 | 目录入口与阅读路径 | 子文档索引、SSOT 声明、依赖关系 |

## 1.4 可选字段

```yaml
标签:
  - Spatial
依赖文档:
  - docs/03_基础服务/06_空间服务/00_总览.md
```

规则：

- 标签：用于检索与聚类，尽量少而准。
- 依赖文档：只写直接依赖，避免堆同领域所有文档。

# 2 目录、路径与命名规范

本节定义“子文档路径与命名规范”的强制口径：读者能根据目录就判断依赖关系，根据文件名就判断排序与引用稳定性。

## 2.1 docs 顶层分层目录（强制）

`docs/` 顶层以依赖顺序分层（00→07），每层下再按子系统拆分目录。每个目录必须包含一个入口文档 `00_总览.md`（`00_文档总览/` 例外，使用 `00_README.md` + `00_总览.md`）。

## 2.2 子系统目录下的“子文档布局”（两种合法形态）

为了兼容已有文档与后续扩写，允许两种布局，但同一目录内必须保持一致。

布局 A（扁平，适合文档少于 6 篇）：

- `00_总览.md`
- `NN_标题.md`

布局 B（按文档类型分组，推荐）：

- `00_总览.md`
- `01_架构设计/NN_标题.md`
- `02_接口规范/NN_标题.md`
- `03_配置结构/NN_标题.md`
- `04_裁决条款/NN_标题.md`
- `05_对齐报告/NN_标题.md`
- `06_测试用例/NN_标题.md`

说明：

- 外部参考不放在业务子系统目录内，统一放在 `docs/07_附录与扩展/`（或引用到 external）。
- “统一规范/技术设计”这类二级主题目录允许存在，但其内部仍需遵循布局 A 或 B。

## 2.3 文件命名（强制）

- 入口文档：`00_总览.md`。
- 同目录的排序文件：`NN_标题.md`，其中 `NN` 为两位数字（`01`～`99`），一经发布尽量不改动，保证引用稳定。
- 兼容入口：保留历史文件名与路径，front-matter `状态: 兼容入口`，正文只写“新位置”，禁止复述正文内容。
- 禁止：空目录、无序号散文件、同目录重复序号、用日期作排序号。

## 2.4 文档标题（强制）

每篇文档的一级标题必须可从标题判断类型与主题，采用以下固定标题模式：

- 总览入口：`# {目录名} 总览`
- 架构设计：`# {系统/模块} 架构设计`
- 接口规范：`# {模块} 接口规范`
- 配置结构：`# {配置名} 配置结构`
- 裁决条款：`# {主题} 裁决条款`
- 对齐报告：`# {主题} 对齐报告`
- 测试用例：`# {用例名} 测试用例`
- 外部参考：`# {项目/库名} 外部参考`

# 3 标题与排版

- 标题层级推荐：`#`、`##`、`###`，不使用四级标题起步。
- 表格用于字段表/差异表/验收矩阵；列名必须稳定。
- 代码块必须标注语言（`csharp`/`json`/`yaml`）。
- 列表行不手写反斜杠加 n 作为换行。

# 4 引用与链接规范

## 4.1 路径口径（强制）

- 引用文档路径使用 `docs/...`（相对 `docs/` 根），并用反引号包裹：`docs/03_基础服务/06_空间服务/00_总览.md`。
- 引用代码路径使用 `src/...`（相对仓库根），并用反引号包裹：`src/Core/Engine/GameEngine.cs`。
- 禁止出现本机绝对路径与 file scheme 的本地链接。

## 4.2 SSOT（强制）

- 同一概念只能有一个权威文档（SSOT）。
- 其它文档只能引用 SSOT，不允许平行叙事导致分叉。
- 当设计文档与实现冲突，必须写对齐报告并链接到 SSOT。

# 5 文档类型分册（按类型拆分）

每种文档类型的“标题与必选章节”模板已拆分到以下分册中（新增文档必须按对应模板写）：
